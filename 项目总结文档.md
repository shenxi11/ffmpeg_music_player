# FFmpeg Music Player 项目总结文档

## 1. 项目概述

- 项目名称：`ffmpeg_music_player`
- 项目定位：基于 Qt + FFmpeg 的桌面端音视频播放器，包含本地播放、在线搜索/下载、歌词展示、插件扩展等能力。
- 当前形态：以 CMake 为主的 C++17 工程，UI 采用 Qt Widgets 与 QML 混合模式，架构正在向 MVVM 逐步迁移。

## 2. 技术栈与依赖

- 语言与框架：C++17、Qt5（Core/Gui/Widgets/Quick/Qml/Network/Multimedia）
- 音视频能力：FFmpeg（avcodec/avformat/avutil/avdevice/swscale/swresample）
- 图形渲染：OpenGL（视频渲染链路）
- 构建系统：CMake 3.16+
- 扩展体系：动态插件（`plugins/`）
- 配套服务：`GoTest/` 提供独立 Go 服务端示例与脚本

## 3. 目录与模块结构（现状）

### 3.1 关键目录

- `qml/`：QML 页面与组件（UI 主体）
- `network/`：新网络层（manager/service/cache/types）
- `viewmodels/`：MVVM 视图模型基础设施
- `plugins/`：插件子项目（音频转换、Whisper 翻译）
- `icon/`：图标资源
- `GoTest/`：Go 后端与启动脚本
- `archive/`：历史/归档代码

### 3.2 关键业务模块（顶层源码）

- 音频链路：`AudioBuffer`、`AudioDecoder`、`AudioPlayer`、`AudioSession`、`AudioService`
- 视频链路：`VideoBuffer`、`VideoDecoder`、`VideoRenderer`、`VideoSession`、`VideoPlayerWindow`
- 页面与交互：`main_widget`、`play_widget`、`music_list_widget_*`、`video_list_widget*`
- 桥接层：`qml_bridge.*` 与一组 `*_qml.h`
- 网络适配：`httprequest_v2.*`（兼容层）+ `network/*`（重构层）

## 4. 已完成的核心工作

### 4.1 播放架构分层

- 音频播放能力已拆分为缓冲、解码、会话、服务等职责单元，降低了 UI 与播放逻辑耦合。
- 视频播放链路与音频链路分离，形成相对独立的解码与渲染路径。

### 4.2 网络层重构

- 新增 `network/` 分层，统一网络请求入口并支持缓存机制。
- 保留 `httprequest_v2` 作为兼容入口，降低旧调用迁移风险。

### 4.3 UI 混合架构与迁移

- 现有系统通过 QWidget 承载主框架，逐步把功能组件迁移到 QML。
- 通过 `qml_bridge` 和 `*_qml.h` 维持 C++ 业务与 QML 视图协同。

### 4.4 插件化能力

- 已具备插件接口与插件管理器。
- 已落地两个插件方向：音频转换、Whisper 翻译。

## 5. 项目亮点

- 功能覆盖较完整：本地音频、在线资源、歌词、视频、下载、插件扩展。
- 架构演进方向明确：从“单体页面逻辑”向“服务化 + MVVM + 组件化”推进。
- 文档沉淀较充分：架构、迁移、优化、问题修复类文档齐全，便于后续维护。

## 6. 当前问题与风险

### 6.1 工程组织复杂度较高

- 根目录源码文件较多，模块边界在目录层不够清晰。
- 历史工程并存（CMake、qmake、VS 工程文件），对新成员理解成本较高。

### 6.2 构建与环境耦合

- CMake 中含本机路径（如 Qt/FFmpeg 默认路径），跨机器复现需要额外配置。
- 工程中存在部分构建产物/中间文件进入源码目录的情况（如插件子目录中的 `CMakeFiles` 痕迹）。

### 6.3 迁移进行中

- MVVM 迁移尚未覆盖全部功能模块。
- Widget 与 QML 双栈并行阶段，维护成本与调试复杂度偏高。

## 7. 阶段性结论

项目已从“可用播放器”进入“可维护工程化”阶段，核心能力已具备，架构重构方向正确。当前最关键工作不是继续堆叠功能，而是提升工程一致性与可维护性，包括目录分层、构建标准化、迁移收口和测试补强。

## 8. 下一阶段建议（可执行）

1. 统一构建入口：以 CMake 为唯一主入口，弱化 qmake/VS 产物在仓库中的存在感。
2. 推进目录重构：按 `core/`、`ui/`、`services/`、`infra/`、`plugins/` 分层落盘。
3. 收口网络调用：新增功能只走 `network/`，逐步移除旧接口依赖。
4. 完成 MVVM 二期：优先迁移播放页、列表页高频交互逻辑。
5. 增加基础测试：先覆盖网络服务、播放状态机、歌词解析等稳定性敏感模块。
6. 清理仓库噪音：移除构建中间文件，补全 `.gitignore` 规则。

## 9. 维护建议

- 变更原则：先文档后改造，先小步迁移再清理旧实现。
- 接口原则：统一通过服务层暴露能力，避免 UI 直接依赖底层模块。
- 发布原则：每次结构调整都附带最小可运行验证（构建 + 启动 + 播放链路）。
