# 网络架构与数据同步（含关键代码说明）

## 1. 文档目标
本文档用于说明客户端当前网络系统的：
- 分层架构与职责边界；
- 请求处理与数据同步链路；
- 下载子系统与主请求链路的关系；
- 关键代码位置，便于后续网络改造与排障。

## 2. 整体分层
当前网络架构可分为 5 层：

1. 业务调用层（UI/Widget）  
入口示例：`loginwidget.cpp:223`、`main_widget.cpp:686`、`video_list_widget.cpp:39`。

2. 业务 API 适配层（`HttpRequestV2`）  
职责：把业务参数转换为接口请求、解析返回、发业务信号。  
代码：`httprequest_v2.h:20`、`httprequest_v2.cpp:194`。

3. 网络服务层（`NetworkService`）  
职责：统一 GET/POST、基础 URL 拼装、全局头、缓存、请求去重映射。  
代码：`network/network_service.h:26`、`network/network_service.cpp:87`。

4. 传输执行层（`NetworkManager`）  
职责：`QNetworkAccessManager` 封装、超时、重试、统计、取消请求。  
代码：`network/network_manager.h:29`、`network/network_manager.cpp:38`。

5. 缓存层（`ResponseCache`）  
职责：GET 缓存、TTL 过期清理、LRU 驱逐。  
代码：`network/response_cache.h:23`、`network/response_cache.cpp:20`。

另外有独立下载通道：
- `DownloadManager` + 内部 `DownloadThread`（线程池并发下载）。  
代码：`download_manager.h:96`、`download_manager.cpp:167`、`download_manager.cpp:25`。

## 4. 请求生命周期（主链路）
以任意业务请求为例，流程如下：

1. UI 调用 `HttpRequestV2` 业务方法。  
例如：`main_widget.cpp:686` 调 `addFavorite`。

2. `HttpRequestV2` 组装 URL、JSON、`RequestOptions`。  
例如：`httprequest_v2.cpp:420`。

3. 进入 `NetworkService::postJson/get`。  
代码：`network/network_service.cpp:43`、`network/network_service.cpp:58`。

4. `NetworkService` 执行缓存判断与请求下发。  
代码：`network/network_service.cpp:87`。

5. `NetworkManager` 创建 `QNetworkRequest`、发起请求、处理超时重试。  
代码：`network/network_manager.cpp:76`、`network/network_manager.cpp:154`、`network/network_manager.cpp:230`。

6. 回调回到 `HttpRequestV2`，解析 JSON，发业务信号。  
例如：`httprequest_v2.cpp:498` 发 `signal_favoritesList`。

7. Widget 订阅信号刷新 UI。  
代码：`main_widget.cpp:730`、`main_widget.cpp:702`。

## 7. 现状问题与后续改造建议
1. `baseUrl` 当前硬编码在 `HttpRequestV2`，建议迁移到 `SettingsManager`/环境变量，支持切换测试与生产环境。  
位置：`httprequest_v2.h:168`。

2. 主请求通道与下载通道并行实现，策略不统一（重试、埋点、鉴权头）。建议把下载也逐步纳入统一网络策略层或共享中间件配置。

3. 已有旧的 `downloadthread.h/.cpp` 与 `download_manager.cpp` 内部 `DownloadThread` 同名并存，建议清理无效实现，避免维护歧义。  
位置：`downloadthread.h:7`、`download_manager.h:67`。

4. 当前请求头未统一注入认证令牌（如 Bearer Token），如果后端升级鉴权，建议通过 `NetworkService::addGlobalHeader` 全局注入。  
位置：`network/network_service.cpp:229`。

5. 建议增加“请求链路ID + 业务ID”的日志字段，把 UI 事件、请求ID、响应ID串起来，便于定位跨模块问题。  
现有请求ID在 `NetworkManager::sendRequest` 已生成：`network/network_manager.cpp:46`。
