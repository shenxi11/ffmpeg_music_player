# 网络架构与数据同步（当前版，含关键代码说明）

## 1. 文档目标
本文档说明客户端当前网络层在 2026-04-08 版本下的真实实现，重点覆盖：

- 请求分层与职责边界；
- 与欢迎页、登录、用户资料、歌单、推荐、视频、歌手搜索相关的主要链路；
- 网络结果如何同步到主窗口与页面；
- 与本地缓存、在线会话、离线模式之间的关系。

## 2. 当前网络分层
当前客户端网络系统可划分为四层：

1. **页面/主窗口触发层**
   - 典型入口：`MainWidget`、`LoginWidget`、`VideoListWidget`、`ServerWelcomeDialog`、`UserProfileWidget`
2. **业务聚合层**
   - 典型入口：`MainShellViewModel`、`LoginViewModel`、`ServerWelcomeViewModel`
3. **业务 API 适配层**
   - `HttpRequestV2`
4. **底层网络执行层**
   - `NetworkService`
   - `NetworkManager`
   - `ResponseCache`

另有一条下载专用链：
- `DownloadManager` + `DownloadThread`
- 它负责大文件下载、进度回调和已下载任务持久化，不与普通 JSON 请求链混用。

## 3. 当前 `HttpRequestV2` 覆盖的业务面
截至当前版本，`HttpRequestV2` 已覆盖以下主要接口能力：

### 3.1 启动与账号
- 欢迎页验证由 `ServerWelcomeViewModel` 负责，调用 `/client/ping` 与 `/client/bootstrap`；
- 账号相关由 `HttpRequestV2` 提供登录、注册、重置密码等接口；
- 登录成功后会回传 `username`、`avatar_url`、`online_session_token`，供主壳层和设置缓存使用。

### 3.2 用户资料
当前已正式接入：
- `getUserProfile(account, sessionToken)`
- `updateUsername(account, sessionToken, username)`
- `uploadAvatar(account, sessionToken, filePath)`

对应结果通过：
- `signalUserProfileResult`
- `signalUpdateUsernameResult`
- `signalUploadAvatarResult`

统一交给 `MainShellViewModel` 处理，并回写 `SettingsManager` 与 `OnlinePresenceManager`。

### 3.3 音乐业务
- 搜索歌曲
- 获取推荐列表、相似推荐、推荐反馈
- 收藏列表与历史列表读取 / 写入 / 删除 / 清空
- 歌手搜索与按歌手获取音乐
- 获取流地址与在线资源信息

### 3.4 私有歌单
当前 playlist 相关接口已形成完整集合：
- 获取歌单列表
- 创建歌单
- 获取歌单详情
- 更新歌单信息
- 删除歌单
- 批量添加歌曲
- 批量移除歌曲
- 歌单排序

### 3.5 视频业务
当前视频业务由 `HttpRequestV2` 提供：
- `getVideoList()`
- `getVideoStreamUrl(videoPath)`

配合 `VideoListViewModel` 和 `VideoPlayerWindow` 形成在线视频播放链。

## 4. 当前主数据同步链

### 4.1 欢迎页验证链
1. `ServerWelcomeDialog` 收集 host/port 或 `host:port` 输入；
2. `ServerWelcomeViewModel` 规范化地址并发起欢迎页验证；
3. 成功时允许进入主窗口，失败时在欢迎页显示错误；
4. 用户也可以直接选择“离线直进”，跳过网络验证进入本地模式。

### 4.2 登录与自动登录链
1. `LoginViewModel` 调用登录接口；
2. 登录成功后，`MainShellViewModel` 保存账号、密码、用户名、头像 URL、在线会话 token；
3. `MainWidget` 依据缓存决定是否触发自动登录；
4. 若当前是离线本地模式，则 `triggerAutoLoginIfNeeded()` 会被直接跳过。

### 4.3 个人资料同步链
1. 个人主页请求刷新资料；
2. `MainShellViewModel::requestUserProfile()` 调用 `HttpRequestV2::getUserProfile()`；
3. 结果返回后，`MainShellViewModel.connections.cpp` 将 `username/avatar_url/created_at/updated_at` 回写 `SettingsManager`；
4. 顶部头像区、个人主页与用户弹窗再统一使用缓存快照刷新。

需要特别说明的是：当前版本已修正“空用户名覆盖本地缓存”的问题，只有当 profile 返回有效非空用户名时，才会覆盖现有缓存用户名。

### 4.4 推荐、收藏、历史、歌单同步链
这些数据当前统一经 `MainShellViewModel` 汇总和分发：
- 页面向主壳层发信号；
- 主壳层调 ViewModel；
- ViewModel 调 `HttpRequestV2`；
- 结果通过对应信号回到 `MainWidget` 和具体列表。

这种方式避免了各页面直接持有 `HttpRequestV2` 或重复实现缓存逻辑。

## 5. 缓存与在线会话
### 5.1 `SettingsManager`
当前持久化的账号与资料缓存已扩展为：
- account / password / remember / auto-login
- cached username
- cached avatar URL
- cached online session token
- cached profile created_at / updated_at

### 5.2 `OnlinePresenceManager`
在线状态管理除了 token 外，现在还能在用户名更新成功后同步更新当前用户名，避免后续在线状态仍携带旧用户名。

### 5.3 `ResponseCache`
GET 请求仍可通过 `ResponseCache` 做 TTL 缓存；但资料修改、收藏写操作、歌单写操作等写请求不会依赖响应缓存，而是由调用方决定后续是否刷新数据。

## 6. 离线模式与网络边界
离线直进模式下，主窗口虽然仍会构建网络相关对象，但在线动作会被统一拦截：
- 在线导航隐藏；
- 搜索禁用；
- 登录、个人主页、推荐、在线音乐、视频等入口不再发请求；
- 业务层保留保护，避免残余调用链误进网络逻辑。

因此当前版本的网络系统既支持完整在线模式，也支持“跳过网络但进入本地播放”的运行形态。

## 8. 当前版本结论
1. 当前客户端网络真相源不是某个页面，而是 `MainShellViewModel` 与 `HttpRequestV2` 的组合；
2. 资料、歌单、推荐、历史、收藏、视频、歌手搜索都已收敛到统一接口层；
3. 欢迎页验证与离线直进已成为启动链的一部分；
4. 在线模式与离线本地模式的边界现在由主窗口统一控制，网络层本身不承担模式判断。
