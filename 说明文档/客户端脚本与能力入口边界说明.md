# 客户端脚本与能力入口边界说明

## 1. 文档目标

这份文档用于明确客户端当前 `Script DSL`、`AgentCapabilityFacade` 和底层 `AgentToolExecutor` 的职责边界，避免服务端在执行载体路由时误判“脚本能做什么”“façade 已经统一到什么程度”“什么仍然必须由服务端先完成”。

目标读者：

- Agent 服务端模板层与执行载体路由层
- 客户端继续推进统一能力入口的开发者
- 联调与排障同学

相关代码锚点：

- `src/agent/script/AgentScriptExecutor.cpp`
- `src/agent/capability/AgentCapabilityFacade.cpp`
- `src/agent/tool/AgentToolExecutor.cpp`
- `src/agent/AgentChatViewModel.cpp`

---

## 2. 当前三层结构

### 2.1 Script DSL

职责：

- 表达一段顺序执行的客户端动作
- 支持 `saveAs`、`$last`、`$steps.alias`
- 提供 `validate`、`dry-run`、`execute`、`cancel`

不负责：

- 条件判断
- 过滤
- 循环
- 歧义消解
- 事务回滚
- 策略路由

结论：

- Script DSL 是“执行载体”
- 不是“任务规划器”

### 2.2 AgentCapabilityFacade

职责：

- 为三类入口提供统一能力入口：
  - 直接 `tool_call`
  - 脚本步骤执行
  - 后续 GUI/自动化入口

当前实现状态：

- 已完成第一刀收束
- 入口已经统一
- 但内部仍主要复用 `AgentToolExecutor`

结论：

- façade 当前是“统一入口外观层”
- 还不是完整领域服务层

### 2.3 AgentToolExecutor

职责：

- 负责把能力名映射到当前客户端真实业务调用
- 直接连接：
  - `MainShellViewModel`
  - `AudioService`
  - 下载、视频、桌面歌词、插件、设置等能力桥接

结论：

- `AgentToolExecutor` 仍是当前阶段的主要底层执行器
- 但不应继续被服务端视作“唯一能力模型”

---

## 3. 当前边界的核心判断

### 3.1 服务端不应把脚本执行能力误判成“客户端已支持完整任务自动化”

当前客户端脚本执行器能做的是：

1. 校验脚本结构
2. 对步骤做最小风险分层
3. 顺序执行步骤
4. 回传步骤级和最终执行结果

不能做的是：

1. 自动补对象
2. 在脚本里按结果筛选候选
3. 发现歧义后自动澄清
4. 在失败后自动回滚
5. 在高风险时自行决定审批

因此：

- 服务端必须先把目标和对象处理到“可执行状态”
- 再决定是否交给脚本

### 3.2 服务端不应把 façade 误判成“底层能力已经完全领域化”

当前 façade 的统一程度：

- 已统一入口
- 尚未统一内部业务实现

这意味着：

- 服务端可以把客户端认知成“有统一能力入口”
- 但不能假设 façade 背后已经具备完整的命令总线、状态仓库、回滚机制

### 3.3 直接工具链并不是失败，而是当前阶段的合理执行载体

如果一个目标：

- 还缺对象
- 或脚本 DSL 无法表达中间决策

那么走 `tool_chain` 是当前正确路径，而不是退化错误。

关键是：

- 服务端要显式知道自己选的是 `tool_chain`
- 客户端日志要能把这条链打印出来

## 4. 脚本适合做什么

### 4.1 适合脚本的任务类型

满足以下条件时，优先考虑脚本：

1. 已有明确对象，或对象可通过前序固定步骤获得
2. 执行路径是线性的
3. 不需要在步骤间做条件判断
4. 后续步骤只依赖前面步骤的结构化结果

典型例子：

- `searchTracks -> playTrack`
- `getRecentTracks -> addToPlaybackQueue`
- `getPlaylistTracks -> setPlaybackQueue -> playAtIndex`
- `createPlaylist -> addPlaylistItems`

### 4.2 当前脚本不适合的任务类型

以下场景当前不应强行脚本化：

1. “查多个候选后按名字或属性挑选”
2. “如果结果为空就走另一条链”
3. “如果歌单名歧义就发澄清”
4. “批量写操作失败后要自动补救”
5. “需要跨域回滚”

典型例子：

- “把我最喜欢的、最近听得最多的三首歌加入流行歌单”
- “如果没有流行歌单就先创建一个”
- “创建歌单并从名字类似的多个来源歌单中选最新的那个”

---

## 5. façade 适合承接什么

当前 façade 适合承接：

1. 稳定原子能力
2. 脚本步骤调度入口
3. 直接工具调用入口
4. 后续 GUI 命令化入口

典型能力域：

- 搜索与歌曲查询
- 播放控制
- 播放队列
- 歌单读写
- 最近播放 / 喜欢
- 下载任务
- 视频控制
- 桌面歌词
- 插件控制
- 设置读写

当前 façade 不负责：

- 用户意图理解
- 任务模板选择
- 对象歧义消解
- 审批决策
- fallback / replan 策略

这些职责应继续留在服务端。

## 6. 当前推荐的执行载体选择规则

### 6.1 走 `single_tool`

适用：

- 原子控制动作
- 原子查询动作

例子：

- `pausePlayback`
- `resumePlayback`
- `stopPlayback`
- `playNext`
- `getCurrentTrack`
- `getRecentTracks`

### 6.2 走 `tool_chain`

适用：

- 当前还缺对象
- 服务端需要在步骤间做对象解析
- 客户端脚本 DSL 无法表达中间决策

例子：

- `getPlaylists -> getPlaylistTracks`
- `searchTracks -> 服务端选候选 -> playTrack`

### 6.3 走 `script`

适用：

- 对象已明确
- 步骤线性
- 无条件分支
- 当前 DSL 可以表达

例子：

- `searchTracks -> playTrack`
- `getPlaylistTracks -> setPlaybackQueue -> playAtIndex`
- `createPlaylist -> addPlaylistItems`

### 6.4 走 `clarification / approval`

适用：

- 目标已理解，但对象歧义
- dry-run 已判高风险
- 需要用户确认

## 7. 当前最重要的服务端误区

### 7.1 误区一：脚本不能立即生成 == 任务失败

错误。

正确理解：

- 当前只是“脚本不是最合适的载体”
- 应继续评估是否转 `tool_chain` 或 `clarification`

### 7.2 误区二：façade 已经等于完整统一能力内核

错误。

正确理解：

- 当前 façade 只是统一入口第一刀
- 底层业务逻辑大头仍在 `AgentToolExecutor`

### 7.3 误区三：直接工具调用一定比脚本“低级”

错误。

正确理解：

- 在对象尚未补齐时，`tool_chain` 反而是更合理、更可解释的载体

## 8. 对联调日志的解释方法

### 8.1 有脚本时

说明：

- 服务端已经完成模板选择与对象补齐
- 当前走的是脚本执行载体

应重点关注：

- `dry_run_script`
- `validate_script`
- `execute_script`
- `script_execution_result`

### 8.2 无脚本但有直接工具链时

说明：

- 服务端没有走脚本
- 当前实际执行载体是 `tool_chain`

这不是客户端协议错误。

应重点关注：

- 为什么服务端没有走脚本
- 这是主动路由，还是脚本 planner 失败后的 fallback

### 8.3 无脚本也无直接工具链时

说明：

- 当前更可能是：
  - 服务端阻断
  - 发了澄清
  - 或纯聊天回复

客户端当前会把这类情况标为：

- `无脚本、无工具链，可能为服务端阻断 / 澄清 / 纯聊天回复`

---

## 9. 后续客户端收束方向

### 9.1 下一步应该继续做什么

1. 继续把高频能力从 `AgentToolExecutor` 下沉到 façade
2. 保持 `tool_call` 和脚本都只走 façade
3. 补更多能力边界文档，给服务端稳定消费

### 9.2 当前不该做什么

1. 不要在客户端脚本层假装支持条件/过滤/事务
2. 不要在 façade 还未沉稳前把更多复杂策略塞进客户端
3. 不要让 GUI 直接绕过 façade 再走一套平行链路

---

## 10. 一句话结论

当前客户端最合理的认知方式是：

**Script 是顺序执行载体，façade 是统一能力入口，ToolExecutor 是当前阶段的底层执行器；模板选择、对象解析、执行载体路由仍主要是服务端职责，客户端不应被误用成“自己会补对象和做任务决策”的智能体。**
