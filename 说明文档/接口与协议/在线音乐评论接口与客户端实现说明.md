# 在线音乐评论接口与客户端实现说明

更新时间：2026-04-17

## 1. 背景与目标

服务端已经落地“在线音乐评论”能力。客户端本次只做接入，不改服务端协议。

本次客户端 v1 目标固定为：

- 以当前播放歌曲详情/控制区作为评论主入口
- 支持查看主评论列表
- 支持查看某条主评论下的平铺回复列表
- 支持发表评论、回复、删除本人评论
- 未登录用户可读，不可写
- 仅支持在线歌曲评论，不支持本地离线歌曲评论

不做的内容：

- 不做楼中楼
- 不做点赞
- 不做评论编辑
- 不做评论审核
- 不改服务端接口

## 2. 服务端已落地接口

### 2.1 获取某首在线歌曲的主评论列表

`GET /music/comments?music_path=...&page=1&page_size=20`

说明：

- `music_path` 必填
- `page` 默认 `1`
- `page_size` 默认 `20`，服务端最大 `100`
- 返回的是“主评论列表”，按创建时间倒序
- 若该歌曲还没有评论，返回 `200`，`items=[]`

典型响应字段：

- `music_path`
- `source`
- `source_id`
- `music_title`
- `artist`
- `cover_art_url`
- `root_comment_count`
- `total_comment_count`
- `page`
- `page_size`
- `items`

其中 `items` 的每一项至少包含：

- `comment_id`
- `user_account`
- `username`
- `avatar_url`
- `content`
- `is_deleted`
- `created_at`
- `reply_count`

### 2.2 获取某条主评论下的回复列表

`GET /music/comments/{comment_id}/replies?page=1&page_size=50`

说明：

- 路径里的 `comment_id` 必须是主评论 ID
- 返回的是该主楼下的平铺回复
- 回复列表按创建时间正序
- `page_size` 默认 `50`，服务端最大 `100`

典型响应字段：

- `root_comment_id`
- `page`
- `page_size`
- `total`
- `items`

回复项字段与评论项基本一致，但可能额外带：

- `root_comment_id`
- `reply_to`

`reply_to` 结构如下：

- `comment_id`
- `user_account`
- `username`

客户端渲染时，如果 `reply_to` 存在，应显示为“回复 @用户名”语义，但服务端并不会把 `@` 直接写进 `content`。

### 2.3 创建主评论

`POST /music/comments`

请求体：

```json
{
  "user_account": "demo_account",
  "session_token": "demo_session_token",
  "music_path": "发如雪/发如雪.mp3",
  "music_title": "发如雪",
  "artist": "周杰伦",
  "content": "这首歌的前奏很好听"
}
```

说明：

- 写接口必须登录
- `music_path` 必须是在线曲目标识
- `music_title`、`artist` 可传可不传，但建议客户端传当前展示值，便于线程初始化时补全元数据

成功响应：

- `success`
- `message`
- `comment_id`
- `comment`

### 2.4 在某条主评论下创建回复

`POST /music/comments/{comment_id}/replies`

请求体：

```json
{
  "user_account": "demo_account",
  "session_token": "demo_session_token",
  "content": "我也觉得这段很强",
  "target_comment_id": 123
}
```

说明：

- 路径中的 `comment_id` 必须是主评论 ID
- `target_comment_id` 可省略
- 若省略，等价于“回复楼主”
- 若填写，则可以回复该主楼下的任意一条回复或主评论
- 即使回复某条回复，数据结构仍然是“主楼 + 楼内平铺回复”，不会形成楼中楼

### 2.5 删除评论

`POST /music/comments/{comment_id}/delete`

请求体：

```json
{
  "user_account": "demo_account",
  "session_token": "demo_session_token"
}
```

说明：

- 只能删除自己发的评论或回复
- 服务端执行软删除
- 删除后 `content` 会变成 `该评论已删除`
- 已删除评论仍可能保留在列表中作为占位，客户端不能把它当成“彻底消失”

## 3. 服务端真实行为与客户端注意事项

### 3.1 只支持在线歌曲

服务端当前只接受两类 `music_path`：

- 服务端曲库里的相对路径，例如 `发如雪/发如雪.mp3`
- Jamendo 虚拟路径，例如 `Mayday [jamendo-1431925].mp3`

不支持：

- Windows 绝对路径，如 `E:\Music\xxx.mp3`
- `file:///` 本地路径
- 客户端私有缓存路径

因此客户端 v1 必须只在“在线歌曲”场景开放评论入口。

### 3.2 根评论是倒序，回复是正序

服务端当前排序规则固定为：

- 主评论：`created_at DESC, id DESC`
- 回复：`created_at ASC, id ASC`

客户端不要自行二次倒序，否则会和服务端分页语义打架。

### 3.3 错误响应目前按纯文本处理

评论接口当前实际错误返回来自 `http.Error(...)`，也就是：

- HTTP 状态码正确
- body 主要是纯文本错误消息

客户端不要假设错误体一定是统一 JSON envelope。对评论接口，错误提示应优先按“状态码 + 文本 body”解析。

### 3.4 删除后计数和列表占位要分开理解

服务端会对线程计数做递减，但已删除主评论仍可能保留在 `items` 里作为占位项。

客户端必须遵守：

- 以 `items` 实际内容渲染列表
- 不要仅根据 `root_comment_count == 0` 就直接隐藏当前页评论项
- `is_deleted == true` 的评论要显示占位态，而不是按普通评论渲染

### 3.5 评论线程不存在不是错误

当歌曲从未产生过评论时，服务端会返回：

- `200`
- `items=[]`
- 正常的歌曲元数据字段

客户端应把它当成“空评论区”，不是请求失败。

### 3.6 登录失效与认证不可用需要区分

常见状态码语义：

- `400`：参数错误、空内容、非在线歌曲、非法回复目标
- `401`：在线会话失效，需要重新登录
- `403`：删除了别人的评论
- `404`：评论或主评论不存在
- `503`：评论服务当前无法校验登录态

其中：

- `401` 应触发登录态补救或登录引导
- `503` 应提示“稍后再试”，不要误导成“账号失效”

## 4. 客户端 v1 接入建议

## 4.1 入口位置

本次评论主入口固定为“当前播放歌曲详情/控制区”。

结合当前代码结构，优先从这些位置落地：

- `src/ui/widgets/playback/play_widget.*`
- `qml/components/playback/ControlBar.qml`
- 如需弹层/面板，可新增 `qml/components/playback/CommentPanel.qml` 或等价组件

不要求 v1 先在在线列表、收藏、历史、歌单卡片上做评论入口全覆盖。

### 4.2 推荐改动层级

网络层：

- `src/network/httprequest_v2.*`
- 新增评论相关请求方法和响应解析

ViewModel 层：

- `src/viewmodels/MainShellViewModel.*`
- 对外暴露：
  - 获取评论线程
  - 获取回复列表
  - 创建主评论
  - 创建回复
  - 删除评论

播放页/UI 层：

- `src/ui/widgets/playback/play_widget.*`
- `qml/components/playback/*`

### 4.3 登录态来源

客户端不要新造认证字段，直接复用现有用户态：

- `MainShellViewModel::currentUserAccount()`
- `MainShellViewModel::currentOnlineSessionToken()`

写接口统一透传：

- `user_account`
- `session_token`

### 4.4 评论入口启用条件

建议按当前播放状态做判断：

- 当前曲目没有有效 `music_path`：禁用
- 当前曲目是本地绝对路径 / `file:///` 路径：禁用
- 当前曲目是服务端相对路径：启用
- 当前曲目是 Jamendo 虚拟路径 `[jamendo-xxx]`：启用

在当前代码里，优先使用当前播放曲目的原始 `music_path` 作为评论线程键，不要用标题或歌手反推。

优先参考：

- `PlaybackViewModel::currentFilePath()`
- `PlayWidget::getNetFlag()`

如果客户端已有“当前在线歌曲原始路径”的缓存，也应优先复用该值。

### 4.5 UI 行为建议

评论面板默认展示：

- 歌曲标题、歌手、封面
- 主评论列表
- 评论输入框

主评论项应支持：

- 展示用户名、头像、时间、内容
- 展示回复数
- 若 `is_deleted == true`，显示 `该评论已删除`
- 若当前用户是评论作者，显示删除入口
- 点击“查看回复”后加载该主楼回复列表

回复列表应支持：

- 展示平铺回复
- 若有 `reply_to`，显示“回复 @用户名”
- 回复别人时只需要记录 `target_comment_id`，不要在客户端拼接成真正楼中楼树

未登录行为：

- 允许打开评论面板和浏览评论
- 点击“发送评论”“回复”“删除”时，提示登录或触发既有登录流程

发送成功后的刷新策略建议：

- 发主评论成功：刷新当前歌曲主评论列表第一页
- 发回复成功：刷新当前主楼回复列表，并同步刷新主评论列表第一页或局部更新对应 `reply_count`
- 删除成功：局部替换为占位态，或重新拉取当前页

## 5. 客户端验收标准

客户端实现完成后，至少满足这些验收项：

1. 在线歌曲打开播放详情后可进入评论面板
2. 本地歌曲评论入口隐藏或不可用
3. 未登录时可以查看评论，但不能发表评论/回复/删除
4. 登录后可以发表主评论
5. 登录后可以回复楼主
6. 登录后可以回复楼内其他回复，且显示 `reply_to`
7. 登录后只能删除自己的评论
8. 已删除评论显示占位文本 `该评论已删除`
9. Jamendo 虚拟路径歌曲也可以正常读取和提交评论
10. 评论接口错误时，客户端能按 HTTP 状态码 + 文本 body 做兜底提示
