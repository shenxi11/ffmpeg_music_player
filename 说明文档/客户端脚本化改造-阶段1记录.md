# 客户端脚本化改造阶段记录

## 阶段 1：最小脚本执行器落地

### 阶段目标
- 在不推翻现有 `AgentToolExecutor` 的前提下，为客户端增加统一脚本入口。
- 先让客户端具备“按步骤顺序执行脚本”的能力。
- 为后续 Agent 端从“单工具调用”切换到“生成脚本并交给客户端执行”打基础。

### 已完成改造

#### 1. 新增脚本执行器模块
- 新增文件：
  - `src/agent/script/AgentScriptExecutor.h`
  - `src/agent/script/AgentScriptExecutor.cpp`
- 角色说明：
  - 负责解析受控 JSON 脚本。
  - 负责校验脚本版本、步骤结构、`saveAs` 命名合法性。
  - 负责顺序执行脚本步骤，并复用现有 `AgentToolExecutor::executeToolCall(...)`。
  - 负责维护每一步结果、`$last` 引用以及 `$steps.<alias>` 引用。

#### 2. 新增脚本 DSL 的最小约束
- 当前支持的脚本结构：

```json
{
  "version": 1,
  "title": "播放周杰伦",
  "steps": [
    {
      "id": "search_jay",
      "action": "searchTracks",
      "args": {
        "keyword": "周杰伦",
        "limit": 5
      },
      "saveAs": "search"
    },
    {
      "id": "play_first",
      "action": "playTrack",
      "args": {
        "trackId": "$steps.search.items.0.trackId"
      }
    }
  ]
}
```

- 当前支持的引用形式：
  - `$last.xxx`
  - `$steps.<saveAs>.xxx`
- 当前约束：
  - 仅支持 `version=1`
  - 仅支持顺序执行
  - 不支持循环、分支、条件判断、并发步骤
  - 参数引用必须是完整字符串，暂不支持字符串内插

#### 3. 把脚本执行器接入 `AgentChatViewModel`
- 修改文件：
  - `src/agent/AgentChatViewModel.h`
  - `src/agent/AgentChatViewModel.cpp`
- 新增能力：
  - `validateClientScript(const QString& scriptText)`
  - `executeClientScript(const QString& scriptText)`
- 当前用途：
  - 先作为客户端内部统一脚本入口。
  - 后续可被协议层、调试入口或测试入口直接复用。

### 本阶段没有做的事
- 还没有新增新的 WebSocket 协议消息类型。
- 还没有让服务端 Agent 直接发送脚本给客户端。
- 还没有把 GUI 全量改为统一脚本/命令入口。
- 还没有做脚本审批、回滚、事务、复杂权限分层。

### 当前阶段结论
阶段 1 已经完成“受控脚本执行器”的最小骨架，客户端具备了：
- 校验脚本
- 顺序执行步骤
- 保存中间结果
- 做基础变量引用
- 复用现有工具执行器

这意味着下一阶段无需从零设计执行器，可以直接开始：
- 扩展脚本 DSL
- 增加协议层 `execute_script`
- 把 GUI 与 Agent 的更多能力收束到统一脚本/命令入口

- `src/agent/script/AgentScriptExecutor.h`
- `src/agent/script/AgentScriptExecutor.cpp`

## 阶段 2：脚本协议入口接入

### 阶段目标
- 把第一阶段的“客户端内部脚本执行器”抬升成正式协议能力。
- 让服务端可以通过 WebSocket 发起：
  - `validate_script`
  - `execute_script`
- 让客户端能够返回：
  - 校验结果
  - 脚本启动事件
  - 步骤事件
  - 最终执行结果

### 已完成改造

#### 1. 协议路由器新增脚本请求解析
- 修改文件：
  - `src/agent/protocol/AgentProtocolRouter.h`
  - `src/agent/protocol/AgentProtocolRouter.cpp`
- 已新增支持的入站消息：
  - `validate_script`
  - `execute_script`
- 当前兼容两种脚本负载形式：
  - `scriptText: "<json-string>"`
  - `script: { ... }`

#### 2. WebSocket 客户端新增脚本结果回传能力
- 修改文件：
  - `src/agent/AgentWebSocketClient.h`
  - `src/agent/AgentWebSocketClient.cpp`
- 已新增出站消息：
  - `script_validation_result`
  - `script_execution_started`
  - `script_step_event`
  - `script_execution_result`

#### 3. 聊天业务层接入脚本协议闭环
- 修改文件：
  - `src/agent/AgentChatViewModel.h`
  - `src/agent/AgentChatViewModel.cpp`
- 新增处理链路：
  - 收到 `validate_script` 后：
    - 调用 `validateClientScript(...)`
    - 通过 `sendScriptValidationResult(...)` 回传结果
  - 收到 `execute_script` 后：
    - 先本地校验
    - 再调用 `executeClientScript(...)`
    - 通过执行器信号持续回传：
      - 启动事件
      - 步骤开始
      - 步骤结束
      - 最终结果

### 当前阶段结论
阶段 2 完成后，客户端脚本能力已经不再只是“内部入口”，而是具备了正式协议闭环：
- 服务端可发脚本校验请求
- 服务端可发脚本执行请求
- 客户端会持续回传脚本执行进度与最终结果

这意味着下一阶段的重点不再是“如何接协议”，而是：
- 统一服务端脚本生成格式
- 增加审批/权限/风险分层
- 扩展 DSL 表达能力

### 本阶段新增协议清单

#### 入站
- `validate_script`
- `execute_script`

#### 出站
- `script_validation_result`
- `script_execution_started`
- `script_step_event`
- `script_execution_result`

## 阶段 3：脚本风险分层与审批门槛

### 阶段目标
- 让客户端不只是“能执行脚本”，还能够判断脚本风险。
- 为服务端 Agent 的自动执行策略提供稳定依据。
- 在客户端侧增加最小审批门槛，避免高风险脚本被无条件执行。

### 已完成改造

#### 1. 脚本校验结果新增策略信息
- 修改文件：
  - `src/agent/script/AgentScriptExecutor.h`
  - `src/agent/script/AgentScriptExecutor.cpp`
- 当前在 `validateScript(...)` 返回中新增：
  - `requiresApproval`
  - `containsWrite`
  - `riskLevel`
  - `autoExecutable`
- 每个步骤当前也会带上 `policy`：
  - `readOnly`
  - `requireApproval`
  - `riskLevel`

#### 2. 脚本步骤开始基于真实工具定义做校验
- 当前 `AgentScriptExecutor` 会复用 `ToolRegistry` 做：
  - 工具存在性校验
  - 参数完整性校验
  - 读/写属性判断
  - 审批需求判断

#### 3. 执行脚本时新增最小审批门槛
- 修改文件：
  - `src/agent/AgentChatViewModel.cpp`
- 当前行为：
  - 如果脚本整体 `requiresApproval=true`
  - 且服务端请求载荷里没有显式 `approved=true`
  - 则客户端直接拒绝执行，并返回：
    - `code=approval_required`

### 当前风险分层规则
- `low`
  - 只读步骤
- `medium`
  - 写操作但不强制审批
- `high`
  - 工具定义中 `requireApproval=true`

### 当前阶段结论
阶段 3 完成后，客户端脚本能力已经具备最小可用的策略层：
- 客户端能判断脚本风险
- 服务端能得到结构化风险信息
- 高风险脚本不会被无条件执行

这一步对 Agent 端很重要，因为它意味着：
- 可以开始基于客户端真实策略做自动执行分层
- 不再只能靠服务端自己猜哪些动作危险

## 阶段 4：脚本取消、超时与执行审计

### 阶段目标
- 让脚本执行具备可控性，避免长链路任务只能被动等待。
- 为服务端 Agent 提供“取消执行”和“超时兜底”的能力。
- 为后续问题排查、回放和审计提供更完整的执行时间字段。

### 已完成改造

#### 1. 脚本新增超时字段
- 修改文件：
  - `src/agent/script/AgentScriptExecutor.h`
  - `src/agent/script/AgentScriptExecutor.cpp`
- 当前脚本支持顶层字段：
  - `timeoutMs`
- 当前约束：
  - 默认值 `60000`
  - 合法范围 `1 ~ 300000`
- 超时后客户端会自动终止脚本执行，并返回：
  - `code=script_timeout`

#### 2. 脚本执行支持取消
- 修改文件：
  - `src/agent/script/AgentScriptExecutor.h`
  - `src/agent/script/AgentScriptExecutor.cpp`
  - `src/agent/protocol/AgentProtocolRouter.h`
  - `src/agent/protocol/AgentProtocolRouter.cpp`
  - `src/agent/AgentWebSocketClient.h`
  - `src/agent/AgentWebSocketClient.cpp`
  - `src/agent/AgentChatViewModel.h`
  - `src/agent/AgentChatViewModel.cpp`
- 新增入站消息：
  - `cancel_script`
- 新增出站消息：
  - `script_cancellation_result`
- 当前行为：
  - 服务端传入 `executionId`
  - 客户端尝试取消对应执行实例
  - 若取消成功，会先回 `script_cancellation_result`
  - 同时原始执行请求对应的 `script_execution_result` 会以失败结束，并带：
    - `code=script_cancelled`

#### 3. 执行报告新增审计字段
- 当前 `script_execution_started` 会带：
  - `startedAt`
  - `timeoutMs`
- 当前 `script_execution_result` 中的 `report` 会带：
  - `status`
  - `startedAt`
  - `finishedAt`
  - `durationMs`

### 当前阶段结论
阶段 4 完成后，客户端脚本协议已经具备最小可用的执行控制能力：
- 可以设置超时
- 可以主动取消
- 可以拿到执行时长与状态

这意味着服务端后续不必再把脚本执行当作“只能发起、不能控制”的黑盒过程。

### 当前新增协议清单

#### 入站
- `cancel_script`

#### 出站
- `script_cancellation_result`

- `src/agent/script/AgentScriptExecutor.cpp`
- `src/agent/protocol/AgentProtocolRouter.h`

## 阶段 5：dry-run 与步骤领域元数据

### 阶段目标
- 让服务端在真正执行脚本前，先拿到一份不落地副作用的 dry-run 结果。
- 让脚本校验结果不只告诉服务端“能不能执行”，还告诉服务端“涉及哪个业务域、哪类写操作、作用对象是什么”。
- 为下一阶段更细粒度审批域、fallback/replan 和统一能力内核迁移提供结构化输入。

### 已完成改造
#### 1. 新增 dry-run 协议闭环
- 当前客户端已支持的脚本相关入站消息：
  - `validate_script`
  - `dry_run_script`
  - `execute_script`
  - `cancel_script`
- 当前客户端已支持的脚本相关出站消息：
  - `script_validation_result`
  - `script_dry_run_result`
  - `script_execution_started`
  - `script_step_event`
  - `script_execution_result`
  - `script_cancellation_result`

#### 2. `AgentChatViewModel` 补齐 dry-run 处理链路
- 修改文件：
  - `src/agent/AgentChatViewModel.cpp`
- 新增能力：
  - `dryRunClientScript(const QString& scriptText)`
  - `onScriptDryRunRequestReceived(...)`
- 当前行为：
  - 收到 `dry_run_script` 后
  - 调用 `AgentScriptExecutor::dryRunScript(...)`
  - 通过 `sendScriptDryRunResult(...)` 回传结果

#### 3. 脚本校验结果新增聚合元数据
- 修改文件：
  - `src/agent/script/AgentScriptExecutor.cpp`
- 当前 `validateScript(...)` / `dryRunScript(...)` 返回中，除原有字段外，还会聚合：
  - `domains`
  - `mutationKinds`
  - `targetKinds`
- 这些字段来自每个步骤的 `policy` 聚合，便于服务端直接做：
  - 领域级审批
  - fallback/replan
  - 任务分层

#### 4. 当前单步策略字段现状
- 每个 `steps[].policy` 当前会带：
  - `tool`
  - `readOnly`
  - `requireApproval`
  - `riskLevel`
  - `domain`
  - `mutationKind`
  - `targetKind`

### 当前阶段结论
阶段 5 完成后，客户端脚本能力已经不只是“可执行”，而是具备了“执行前预演 + 领域识别”的最小能力模型：
- 服务端可先 `dry_run_script`，再决定是否真正执行
- 服务端可根据 `domains / mutationKinds / targetKinds` 做更细粒度审批和策略分流
- 当前脚本执行仍然基于顺序步骤，但已经具备向“统一能力内核”迁移所需的策略元数据

## 阶段 6：统一能力入口第一刀

### 阶段目标
- 不再让脚本执行器和聊天调度层直接依赖 `AgentToolExecutor`。
- 在现有代码不大拆的前提下，先增加一层正式的统一能力入口，作为后续 `CapabilityFacade` 演进的起点。
- 为后续 GUI、脚本、Agent 逐步共用一套业务入口铺路。

### 已完成改造
#### 1. 新增 `AgentCapabilityFacade`
- 新增文件：
  - `src/agent/capability/AgentCapabilityFacade.h`
  - `src/agent/capability/AgentCapabilityFacade.cpp`
- 当前职责：
  - 统一承接能力调用
  - 向外暴露：
    - `setMainShellViewModel(...)`
    - `executeCapability(...)`
    - `toolRegistry()`
  - 当前阶段底层仍复用 `AgentToolExecutor`

#### 2. `AgentToolExecutor` 新增注册表透出接口
- 修改文件：
  - `src/agent/tool/AgentToolExecutor.h`
- 新增：
  - `toolRegistry() const`
- 当前用途：
  - 让 façade 和脚本执行器共享同一份工具注册定义
  - 避免脚本层和工具层各自维护一份工具元数据

#### 3. `AgentChatViewModel` 改为通过 façade 分发能力
- 修改文件：
  - `src/agent/AgentChatViewModel.h`
  - `src/agent/AgentChatViewModel.cpp`
- 当前变化：
  - `m_toolExecutor` 改为 `m_capabilityFacade`
  - `onToolCallReceived(...)` 不再直接调用 `AgentToolExecutor::executeToolCall(...)`
  - 改为调用 `AgentCapabilityFacade::executeCapability(...)`
  - 工具结果和进度信号也改为由 façade 回传

#### 4. `AgentScriptExecutor` 改为依赖 façade
- 修改文件：
  - `src/agent/script/AgentScriptExecutor.h`
  - `src/agent/script/AgentScriptExecutor.cpp`
- 当前变化：
  - 构造函数参数改为 `AgentCapabilityFacade*`
  - 步骤执行时改为调用 `executeCapability(...)`
  - 工具注册表改为优先从 façade 读取

### 当前阶段结论
阶段 6 完成后，客户端虽然还没有做到完整统一能力内核，但已经迈出了关键一步：
- 原始 `tool_call`
- 脚本步骤执行

这两条链路现在都先经过 `AgentCapabilityFacade`，不再各自直接碰 `AgentToolExecutor`。

这意味着后续如果要继续推进：
- GUI 也接入同一层
- 把部分业务逻辑从 `AgentToolExecutor` 下沉到 façade

都已经有明确的承接点了。
