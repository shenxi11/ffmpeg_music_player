# 音频架构与内部数据同步（当前版，含关键代码说明）

## 1. 文档目标
本文档说明当前版本音频播放系统的真实结构，重点回答三件事：

1. 音频播放核心类如何分工；
2. 当前播放状态、播放队列、封面和歌手等数据是如何同步到 UI 的；
3. 本地与在线音乐在当前版本中如何共享播放链但保持不同的数据补偿策略。

## 2. 核心模块
| 模块 | 代码位置 | 职责 |
|---|---|---|
| `AudioService` | `src/audio/AudioService.*` | 音频服务入口，管理播放列表、当前索引、播放控制与会话生命周期 |
| `AudioSession` | `src/audio/AudioSession.*` | 单次音频会话编排层，负责解码、缓冲、接管音频输出 |
| `AudioDecoder` | `src/audio/AudioDecoder.*` | 基于 FFmpeg 的音频解封装、解码与重采样 |
| `AudioPlayer` | `src/audio/AudioPlayer.*` | 音频输出与播放时钟维护 |
| `PlaybackViewModel` | `src/viewmodels/PlaybackViewModel.*` | 当前播放真相源，向 UI 暴露标题、歌手、封面、时长、进度、队列快照 |
| `PlayWidget` | `src/ui/widgets/playback/play_widget.*` | 播放详情页与底部主控件联动层 |
| `LocalMusicCache` | `src/download/local_music_cache.*` | 本地歌曲封面、时长、歌手等持久化缓存 |
| `cover_lookup` | `src/common/cover_lookup.*` | 队列、历史、下载页的封面记忆查询辅助模块 |

## 3. 当前播放主链
### 3.1 控制链
1. 页面或列表动作发起播放、暂停、切歌、跳转进度；
2. `PlaybackViewModel` 接受命令并调用 `AudioService`；
3. `AudioService` 管理播放列表，必要时创建或切换 `AudioSession`；
4. `AudioSession` 调用 `AudioDecoder` 解码并把 PCM 交给 `AudioPlayer`；
5. `AudioPlayer` 输出音频，同时回推位置与时钟变化；
6. `AudioService` 再将状态、位置、时长与元数据转发给 `PlaybackViewModel`；
7. `PlaybackViewModel` 更新 UI 属性，驱动播放页、列表页和播放队列刷新。

### 3.2 数据链
当前版本里，播放状态相关的 UI 不再各自维护一套状态，而是统一读取 `PlaybackViewModel`：

- `currentTitle`
- `currentArtist`
- `currentAlbumArt`
- `currentUrl`
- `isPlaying`
- `duration`
- `position`
- `playlistSnapshot`

其中 `playlistSnapshot` 是当前版本非常关键的统一结构，它用于：
- 右侧播放队列展示；
- 当前曲目标记；
- 非当前歌曲封面、标题、歌手的补偿；
- 切歌后上一首封面不回退到默认图。

## 4. 当前版本的数据同步重点

### 4.1 队列快照与元数据缓存
`PlaybackViewModel` 维护了播放队列元数据缓存，并通过 `rememberTrackMetadata(...)` 与 `cacheCurrentTrackMetadata()` 持续补齐：

- 入队时先记住 songData；
- 当前歌曲播放中若拿到新的标题、歌手、封面，会回写缓存；
- `playlistSnapshot()` 按“当前播放态 -> 队列缓存 -> 本地/封面查询 -> 默认值”的优先级生成统一快照。

这意味着当前队列与主播放页不再是两条松散链，而是共享一份更稳定的元数据来源。

### 4.2 本地音乐的元数据持久化
本地歌曲当前支持在播放过程中回写三类数据到 `LocalMusicCache`：

- `coverUrl`
- `duration`
- `artist`

具体做法是：
- `PlayWidget` 在 `currentArtistChanged`、`currentAlbumArtChanged`、`durationChanged` 等时机统一触发 `emitLocalMetadataUpdateIfNeeded()`；
- 只要当前播放的是本地文件，就向外发出 `signalMetadataUpdated(filePath, coverUrl, duration, artist)`；
- `MainWidget` 将该信号同时转发给本地列表与下载列表；
- `MainShellViewModel::updateLocalMusicCacheMetadata(...)` 最终写入 `LocalMusicCache`。

这样一来，本地列表中的“未知艺术家”在播放时若能解析到真实歌手，就会被永久记住；下次打开应用仍然显示真实歌手名。

### 4.3 下载列表的封面与歌手补偿
当前下载页不再只依赖下载任务最初写入的 `coverUrl`。现在的优先级是：

1. `DownloadTask.coverUrl`
2. `LocalMusicCache` 中按 `savePath` 记住的封面/歌手/时长
3. 默认封面

而且播放下载歌曲后拿到的新元数据，也会通过统一回写链同步到下载模型并持久化到 `DownloadManager`，因此：
- 当前会话里列表能立刻更新；
- 重启后封面和歌手仍可保留。

### 4.4 列表封面动作统一
当前列表页的封面交互统一采用 `SongCoverAction`：
- 悬浮非当前歌曲时显示播放按钮；
- 当前歌曲未悬浮时显示播放中指示器；
- 当前歌曲悬浮时显示暂停按钮。

这一行为已经在本地列表、下载列表、喜欢列表、历史列表、在线列表等主要页面上对齐，底层依赖的仍是 `PlaybackViewModel` 的统一播放状态，而不是页面各自猜测。

## 5. 关键文件索引

- 播放真相源：`src/viewmodels/PlaybackViewModel.cpp`

- 下载页元数据模型：`src/download/download_task_model.cpp`
- 封面记忆辅助：`src/common/cover_lookup.cpp`

## 6. 当前版本结论
1. 音频播放的真正控制核心仍是 `AudioService` + `AudioSession`；
2. UI 层的唯一播放状态真相是 `PlaybackViewModel`；
3. 队列快照、封面补偿、歌手回写、本地缓存持久化已经形成闭环；
4. 本地、下载、历史、在线等不同来源的歌曲共享同一播放链，但在元数据补偿和路径解析上各有针对性兜底。

## 7. 与 3 月版本相比的变化
- 增强了 `playlistSnapshot` 的作用，不再只是简单队列投影；
- 新增 `cover_lookup` 统一封面记忆查询；
- 本地元数据回写增加了 `artist`，解决“未知艺术家”长期不更新的问题；
- 下载列表同步接入播放后元数据回写和本地缓存兜底；
- 列表封面交互统一收敛到 `SongCoverAction`。
