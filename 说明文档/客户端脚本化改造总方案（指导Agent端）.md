# 客户端脚本化改造总方案（当前版）

## 1. 目标
客户端脚本化改造的目标是：让服务端 Agent 能以稳定、可约束、可审计的方式调用客户端能力，而不是直接依赖 UI 自动化。

## 2. 当前总方案结构
当前仍采用三层结构：
1. 工具声明层：`ToolRegistry`
2. 能力执行层：`AgentCapabilityFacade` + `AgentToolExecutor`
3. 顺序脚本层：`AgentScriptExecutor`

## 3. 当前版本应遵守的原则
- 先复用主窗口和 ViewModel 现有业务入口，再暴露为 Agent 工具；
- 工具设计以“当前代码可稳定执行”为前提，不以未来设想充当当前能力；
- 离线模式、资料页、下载补偿等当前版本新增主链，必须在能力边界文档中明确说明是否开放给 Agent。

## 4. 当前最适合继续扩展的方向
- 搜索与歌单的复合任务编排；
- 播放队列查询与操作的一致性增强；
- 对当前用户资料页能力进行审慎开放；
- 基于文档校准后的 RAG 检索支持更稳定的工具路由。

## 5. 当前不建议直接推进的方向
- 让客户端脚本层自己做复杂对象解析；
- 把欢迎页模式切换、登录态恢复等高副作用流程全部暴露为无保护工具；
- 在没有统一状态真相的页面上直接新增 Agent 入口。
