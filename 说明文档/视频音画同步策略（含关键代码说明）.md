## 3. 同步总策略

当前实现是**音频主时钟**策略：

1. 音频播放位置作为基准时钟。
2. 视频渲染时比较“下一帧 PTS”和“音频时钟”。
3. 视频超前就等待；视频落后就丢帧追赶。
4. seek 后通过首个有效音频帧 PTS 重建时钟锚点。

## 4. 关键代码与解释

## 4.1 主时钟选择逻辑
代码位置：`MediaSession.cpp:450`

```cpp
qint64 MediaSession::getCurrentPosition() const
{
    if (m_audioCodecCtx) {
        const auto& player = AudioPlayer::instance();
        if (player.writeOwner() == m_audioWriteOwnerId && player.isPlaying()) {
            return player.getPlaybackPosition();
        }
    }

    if (m_videoSession) {
        qint64 videoPTS = m_videoSession->getCurrentPTS();
        if (videoPTS > 0) {
            return videoPTS;
        }
    }

    return m_masterClock;
}
```

解释：

- 第一优先级是音频时钟（且 owner 匹配）。
- 音频不可用时退化到视频 PTS。
- 再不行使用 `m_masterClock` 兜底。

这保证了播放中的主时钟稳定性，避免不同链路来回跳。

## 4.2 Seek 时的缓冲清理与重建
代码位置：`MediaSession.cpp:335`

```cpp
void MediaSession::seekTo(qint64 positionMs)
{
    // ...停止 demux 线程...
    av_seek_frame(m_formatContext, -1, timestamp, AVSEEK_FLAG_BACKWARD);

    if (m_audioCodecCtx) {
        avcodec_flush_buffers(m_audioCodecCtx);
        AudioPlayer::instance().resetBuffer();
        m_seekPending = true;
    }

    if (m_videoSession) {
        m_videoSession->flush();
    }

    m_masterClock = positionMs;
    // ...重启 demux...
}
```

解释：

- seek 后会同时清空音频与视频缓冲，防止旧时间轴数据混入。
- `m_seekPending=true` 表示“下一帧音频到来时，需要重锚定时钟”。

## 4.3 Seek 完成后的时钟重锚定
代码位置：`MediaSession.cpp:693`

```cpp
if (m_seekPending) {
    AudioPlayer::instance().setCurrentTimestamp(inputPtsMs);
    m_seekPending = false;
    qDebug() << "[MediaSession] Seek completed - clock synced to actual PTS:" << inputPtsMs << "ms";
}
```

解释：

- seek 后不直接盲设最终时钟，而是等待第一帧有效音频 PTS。
- 这样能把播放时钟对齐到“真实解码后的时间点”，减少 seek 后漂移。

## 4.4 渲染器中的核心音画对齐逻辑
代码位置：`VideoRendererGL.cpp:708`

```cpp
qint64 audioClock = 0;
if (AudioPlayer::instance().isPlaying()) {
    audioClock = AudioPlayer::instance().getPlaybackPosition();
} else {
    audioClock = m_lastPTS;
}

VideoFrame* frame = m_buffer->peek();
if (!frame) return;

if (frame->pts > audioClock + 100) {
    return; // 视频超前：等待
}

while (frame && audioClock > frame->pts + 200) {
    m_buffer->pop();
    delete frame;
    frame = m_buffer->peek(); // 视频落后：丢帧追赶
}
```

解释：

- 使用音频时钟进行视频调度。
- 阈值策略：
  - `+100ms`：视频超前等待
  - `+200ms`：视频落后丢帧
- 这就是当前音画同步实际生效的主逻辑。

## 4.5 音频时钟如何计算
代码位置：`AudioPlayer.cpp:491`

```cpp
qint64 AudioPlayer::getPlaybackPosition() const
{
    if (!m_isPlaying) return 0;
    if (m_isPaused) return m_pausedPosition;

    const double rate = m_playbackRate.load();
    qint64 scaledElapsed = static_cast<qint64>(m_playbackTimer.elapsed() * rate);
    return m_playbackStartTimestamp + scaledElapsed;
}
```

解释：

- 运行态下，音频位置 = 起始时间戳 + 实际经过时间（考虑倍速）。
- 该值直接被视频渲染器作为同步基准。

## 6. 小结

当前音画同步是“**AudioPlayer 主时钟 + VideoRendererGL 跟随追赶**”：

- 稳定点：结构清晰、seek 后有时钟重建、切换场景可复用。
- 风险点：主动纠偏（`syncAudioVideo()`）尚未完全落地，阈值为固定值，复杂网络场景仍需调参。

后续若要继续提升，可在 `MediaSession::syncAudioVideo()` 中接入真实 `videoPTS`，并将阈值动态化。
