# 服务端第二期改造计划

## 1. 文档目标

这份文档用于定义服务端从“第一期直连工具版”继续升级到“可规划、可审批、可审计的音乐 Agent 第二期”时的实施方向。

第一期已经完成：

- 普通聊天
- 流式输出
- 会话与消息存储
- 工具模式 MVP
- `tool_call / tool_result / clarification_request`
- 低风险工具：
  - `searchTracks`
  - `getCurrentTrack`
  - `getPlaylists`
  - `playTrack`
  - `playPlaylist`

第二期的目标不是简单增加更多工具，而是引入真正可控的“计划执行”能力，让 Agent 能完成写操作，同时保持安全边界和可回溯性。

---

## 2. 第二期目标

第二期主要覆盖四类能力：

- 写操作工具接入
- 计划生成与步骤执行
- 审批机制
- 内部事件审计与状态恢复

核心用户场景：

- “创建一个学习歌单”
- “把最近常听的歌加到学习歌单里”
- “把这首歌加入我喜欢”
- “把刚才搜到的前 3 首加入夜跑歌单”
- “取消收藏这两首歌”

成功标准：

- Agent 能先生成结构化计划，而不是直接执行所有写操作
- 中高风险任务会触发 `plan_preview` 或 `approval_request`
- Qt 端返回 `tool_result` 后，服务端能继续按步骤推进
- 任一步骤失败时，服务端能停在可恢复状态，而不是整条链路丢失
- 所有执行步骤和审批结果都能被审计

---

## 3. 第二期范围

### 3.1 新增工具

建议第二期新增以下工具：

- `createPlaylist`
- `addTracksToPlaylist`
- `favoriteTracks`
- `unfavoriteTracks`
- `getRecentTracks`
- `getTopPlayedTracks`
- `getPlaylistTracks`

其中：

- `getRecentTracks`、`getTopPlayedTracks`、`getPlaylistTracks` 主要用于让计划更可靠
- `createPlaylist`、`addTracksToPlaylist`、`favoriteTracks`、`unfavoriteTracks` 是第二期核心写操作

### 3.2 新增协议能力

建议启用以下消息类型：

- `plan_preview`
- `approval_request`
- `approval_response`
- `progress`
- `final_result`

第一期已有但要继续保留：

- `session_ready`
- `user_message`
- `assistant_start`
- `assistant_chunk`
- `assistant_final`
- `tool_call`
- `tool_result`
- `clarification_request`
- `error`

### 3.3 暂不进入第二期主路径

这些能力建议继续后置：

- 删除歌单
- 批量删除歌曲
- 覆盖歌单内容
- 文件系统读写
- 下载管理
- 多计划并发执行

## 4. 服务端架构演进

### 4.1 从工具直连升级为计划驱动

第一期大致是：

```text
user_message
-> intent
-> maybe tool_call
-> tool_result
-> assistant reply
```

第二期建议升级为：

```text
user_message
-> ParseIntent
-> ResolveContext
-> BuildPlan
-> RiskCheck
-> PlanPreview / Approval
-> ExecutePlanStep
-> HandleToolResult
-> ContinueOrStop
-> FinalResult
```

### 4.2 推荐状态节点

建议服务端运行图增加以下节点：

1. `ParseIntent`
2. `ResolveContext`
3. `BuildPlan`
4. `ValidatePlan`
5. `RiskCheck`
6. `WaitApproval`
7. `ExecutePlanStep`
8. `WaitToolResult`
9. `SummarizeOutcome`
10. `PersistExecutionState`

### 4.3 工作记忆升级

当前工作记忆建议从“候选结果 + pending tool”升级为：

- `intent`
- `entities`
- `ambiguities`
- `workflowMode`
- `pendingToolCall`
- `pendingClarification`
- `activePlan`
- `activePlanStepIndex`
- `approvalStatus`
- `recentTrackCandidates`
- `recentPlaylistCandidates`
- `lastToolResult`
- `lastExecutionError`

说明：

- 第一版仍然可以先放在进程内存中
- 但第二期建议开始设计“可选持久化”
- 至少要支持服务端进程意外退出后的最低限度恢复

## 5. 计划模型建议

### 5.1 计划对象

```json
{
  "planId": "plan-123",
  "sessionId": "session-1",
  "summary": "创建歌单并添加最近常听歌曲",
  "riskLevel": "medium",
  "requiresApproval": true,
  "status": "draft",
  "steps": [
    {
      "stepId": "step-1",
      "tool": "createPlaylist",
      "title": "创建歌单 学习歌单",
      "args": { "name": "学习歌单" },
      "status": "pending"
    },
    {
      "stepId": "step-2",
      "tool": "getTopPlayedTracks",
      "title": "获取最近常听歌曲",
      "args": { "limit": 20 },
      "status": "pending"
    },
    {
      "stepId": "step-3",
      "tool": "addTracksToPlaylist",
      "title": "将歌曲加入歌单",
      "args": {
        "playlistId": "$step-1.playlist.playlistId",
        "trackIds": "$step-2.items[*].trackId"
      },
      "status": "pending"
    }
  ]
}
```

### 5.2 步骤状态

每个 step 建议有稳定状态：

- `pending`
- `waiting_tool`
- `succeeded`
- `failed`
- `skipped`
- `cancelled`

### 5.3 变量引用

第二期建议只支持最小变量替换，不要一开始做复杂表达式引擎。

允许：

- `$step-1.playlist.playlistId`
- `$step-2.items[*].trackId`

不建议第二期做：

- 任意脚本表达式
- 动态条件分支脚本
- 用户自定义模板

---

## 6. 风险分级与审批

### 6.1 风险级别建议

- `low`
  - 搜索
  - 读取状态
  - 播放控制
- `medium`
  - 创建歌单
  - 收藏单首或少量歌曲
- `high`
  - 批量写操作
  - 歌单批量改动
  - 批量收藏或取消收藏

### 6.2 审批策略

建议策略如下：

- `low`
  - 直接执行
- `medium`
  - 先发送 `plan_preview`
  - 客户端可让用户一键继续
- `high`
  - 必须发送 `approval_request`
  - 未收到明确批准，不执行任何写操作

### 6.3 典型审批场景

- 创建歌单并批量加入歌曲
- 批量收藏
- 批量取消收藏
- 向已有歌单写入多首歌曲

## 7. 第二期协议扩展建议

### 7.1 `plan_preview`

服务端 -> Qt

```json
{
  "type": "plan_preview",
  "planId": "plan-123",
  "sessionId": "session-1",
  "summary": "创建学习歌单并添加最近常听歌曲",
  "riskLevel": "medium",
  "steps": [
    { "stepId": "step-1", "title": "创建歌单 学习歌单" },
    { "stepId": "step-2", "title": "获取最近常听歌曲" },
    { "stepId": "step-3", "title": "将歌曲加入歌单" }
  ]
}
```

### 7.2 `approval_request`

服务端 -> Qt

```json
{
  "type": "approval_request",
  "planId": "plan-123",
  "sessionId": "session-1",
  "message": "即将创建歌单并批量添加 20 首歌曲，是否继续？",
  "riskLevel": "high"
}
```

### 7.3 `approval_response`

Qt -> 服务端

```json
{
  "type": "approval_response",
  "planId": "plan-123",
  "approved": true
}
```

### 7.4 `progress`

服务端 -> Qt

```json
{
  "type": "progress",
  "planId": "plan-123",
  "stepId": "step-2",
  "message": "已创建歌单，正在获取最近常听歌曲"
}
```

### 7.5 `final_result`

服务端 -> Qt

```json
{
  "type": "final_result",
  "planId": "plan-123",
  "sessionId": "session-1",
  "ok": true,
  "summary": "已创建歌单“学习歌单”，并添加 20 首歌曲"
}
```

## 8. 服务端实现拆分建议

### 8.1 新模块建议

建议新增这些模块：

- `plan_models.py`
- `plan_builder.py`
- `approval_manager.py`
- `execution_runtime.py`
- `execution_store.py`
- `event_log.py`

### 8.2 各模块职责

`plan_models.py`

- 定义 `Plan`、`PlanStep`、`RiskLevel`

`plan_builder.py`

- 把结构化意图转成最小可执行计划

`approval_manager.py`

- 判断是否需要 `plan_preview`
- 判断是否必须 `approval_request`
- 挂起与恢复执行

`execution_runtime.py`

- 顺序推进步骤
- 解析 step 间变量引用
- 发送 `tool_call`
- 处理 `tool_result`

`execution_store.py`

- 保存 active plan 和 step 状态

`event_log.py`

- 记录内部执行事件
- 便于审计和恢复

## 9. 存储建议

### 9.1 第二期建议新增内部事件存储

第一期会话表和消息表主要服务 UI。

第二期建议新增“内部执行事件”表或日志流，至少记录：

- `eventId`
- `sessionId`
- `planId`
- `stepId`
- `type`
- `payload`
- `createdAt`

事件类型示例：

- `intent_parsed`
- `plan_built`
- `plan_preview_sent`
- `approval_requested`
- `approval_received`
- `tool_call_sent`
- `tool_result_received`
- `step_succeeded`
- `step_failed`
- `plan_finished`

### 9.2 是否需要 SQLite 落表

建议：

- `activePlan` 先支持 SQLite 落表
- 内部事件可以先落 SQLite，也可以先落 NDJSON 文件日志

如果希望支持进程意外中断恢复，至少要把这两类信息持久化：

- 计划本体
- 当前步骤进度

## 10. Qt 端需要配合的点

第二期需要 Qt 端新增这些能力：

- 支持 `plan_preview`
- 支持 `approval_request`
- 支持 `approval_response`
- 支持 `progress`
- 支持 `final_result`
- 接入第二批工具

Qt 端最少要能做到：

- 把计划卡片渲染出来
- 让用户批准或拒绝
- 展示步骤进度
- 在执行完成后展示结果摘要

---

## 11. 推荐实施顺序

### 阶段 A：计划框架先落空骨架

- 定义 `Plan` 数据结构
- 增加 `plan_preview / approval_request / approval_response`
- 先不接复杂工具

目标：

- 让 Qt 端先联调新协议

### 阶段 B：接入第二批只读工具

- `getRecentTracks`
- `getTopPlayedTracks`
- `getPlaylistTracks`

目标：

- 给后续计划提供可靠输入

### 阶段 C：接入写操作工具

- `createPlaylist`
- `addTracksToPlaylist`
- `favoriteTracks`
- `unfavoriteTracks`

目标：

- 实现真正的第二期业务价值

### 阶段 D：加审计和恢复

- 内部事件日志
- active plan 持久化
- 工具超时恢复

目标：

- 让执行链路更稳

---

## 12. 第二期测试重点

### 12.1 正向场景

- 创建空歌单
- 创建歌单并添加最近常听歌曲
- 把最近搜索结果前 3 首加入歌单
- 收藏单首歌曲
- 取消收藏多首歌曲

### 12.2 失败场景

- 歌单已存在
- 目标歌单不存在
- `trackIds` 中有无效项
- 审批被拒绝
- 某一步工具超时
- 工具部分成功部分失败

### 12.3 恢复场景

- 服务端重启后发现有未完成计划
- Qt 端重连后继续等待 `tool_result`
- 用户取消计划后服务端正确清理状态

---

## 13. 一句话结论

第二期的关键不是“再多接几个工具”，而是把后端从“能发工具调用”升级成“能先规划、再审批、再顺序执行、还能审计和恢复”的真正 Agent 执行内核。
