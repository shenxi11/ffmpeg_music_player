# Qt 音乐软件 Agent 提示文档

## 文档目的

这份文档用于总结当前关于 Qt 音乐软件 Agent 的设计思路，并作为后续继续讨论、实现原型、编写提示词、设计协议和拆分模块时的基础输入。

目标不是做一个“会聊天的 AI”，而是做一个：

- 前端入口在音乐软件内部
- 能理解自然语言需求
- 能把需求拆成结构化流程
- 能让用户审阅和修改流程
- 最终通过主程序暴露的业务 API 执行操作

---

## 一、产品目标

希望在 Qt 音乐软件中内置一个 AI 助手，支持用户通过自然语言完成以下操作：

- 创建歌单
- 播放某首歌曲或某个歌单
- 将歌曲加入“喜欢”
- 批量添加歌曲到歌单
- 根据条件筛选和整理歌曲
- 在复杂操作前展示计划并允许用户修改

这个 AI 助手应当像软件里的一个“智能控制台”，而不是一个直接模拟点击界面的机器人。

---

## 二、核心设计原则

### 1. AI 不直接操作 UI

AI 不应直接点击按钮、操作列表控件或模拟用户界面行为。

正确方式是：

- AI 负责理解需求和生成计划
- 主程序负责执行业务动作

例如，不做：

- 找到“喜欢”按钮并点击

而做：

- `favoriteTracks(trackIds)`

### 2. 先规划，后执行

所有复杂任务应优先经过：

```text
自然语言 -> 语义拆解 -> 结构化计划 -> 用户确认/修改 -> 执行
```

### 3. 语义拆解比 RAG 更重要

对于当前音乐软件场景，最重要的能力不是知识检索，而是：

- 意图识别
- 实体抽取
- 指代消解
- 歧义识别
- 结构化流程生成

RAG 可以作为辅助能力，但不是第一优先级。

### 4. 流程必须可纠错

AI 生成的计划不能直接视为最终正确结果，必须支持：

- 预览
- 确认
- 修改参数
- 删除步骤
- 增加步骤
- 再执行

---

## 三、整体架构建议

推荐采用：

```text
Qt 主程序 + Qt 内部 AI 入口 + Agent Sidecar + Host API
```

### 架构角色

#### 1. Qt 主程序

负责：

- 播放控制
- 歌单管理
- 收藏管理
- 曲库搜索
- 当前状态提供

#### 2. Qt 内部 AI 入口

入口在音乐软件内部，例如：

- 侧边栏 AI 面板
- 聊天窗口
- 智能搜索模式
- 操作确认窗口

负责：

- 接收用户输入
- 展示 AI 计划
- 允许用户编辑和确认
- 展示执行进度和结果

#### 3. Agent Sidecar

建议作为独立进程运行。

负责：

- 自然语言理解
- 语义拆解
- 上下文管理
- 计划生成
- 工具编排
- 等待用户确认
- 汇总结果

#### 4. Host API

主程序对 Agent 暴露的标准业务接口。

例如：

- `searchTracks`
- `getCurrentTrack`
- `getRecentTracks`
- `getTopPlayedTracks`
- `createPlaylist`
- `addTracksToPlaylist`
- `playTrack`
- `playPlaylist`
- `favoriteTracks`
- `unfavoriteTracks`

---

## 四、为什么建议 Agent 用 Sidecar 进程

推荐让 Agent 作为独立进程，而不是直接嵌入 Qt 主进程。

典型形态：

```text
MyPlayer.exe
+ AiAssistantPlugin.dll
+ music-agent.exe
```

优点：

- Qt/C++ 与 Python/LangGraph 运行时解耦
- Agent 崩溃不会直接拖死主程序
- 便于升级和替换模型框架
- 更适合插件化和调试
- 更适合未来独立演进

用户感知上，AI 入口依然在你的音乐软件里面。

---

## 五、通信建议

推荐：

- 通信方式：`WebSocket`
- 消息格式：`JSON`

这是因为 Agent 与主程序之间不是单次请求，而是持续的双向交互：

- 用户发起请求
- Agent 分析
- Agent 生成计划
- Agent 请求主程序执行工具
- 主程序返回结果
- Agent 继续下一步
- 最终返回总结

### 推荐消息类型

- `user_message`
- `plan_preview`
- `approval_request`
- `approval_response`
- `tool_call`
- `tool_result`
- `progress`
- `final_response`
- `error`

### 推荐基础字段

每条消息建议包含：

- `type`
- `requestId`
- `sessionId`
- `timestamp`
- `payload`

## 六、Agent 的核心模块设计

### 1. SemanticParser

负责语义拆解，是整个系统最重要的模块。

输出内容至少包括：

- `intent`
- `entities`
- `references`
- `missingFields`
- `ambiguities`
- `riskLevel`

### 2. ContextManager

负责拼装模型调用所需上下文，而不是把所有历史直接塞给模型。

建议上下文分为：

- 系统规则
- 最近对话
- 当前任务状态
- 当前软件状态
- 最近工具结果
- 用户偏好

### 3. PlanBuilder

根据语义拆解结果生成结构化计划。

### 4. PlanValidator

负责：

- 校验工具是否合法
- 校验参数是否完整
- 判断是否需要确认
- 拦截高风险操作

### 5. PlanEditor

负责让用户修改 AI 生成的流程。

支持：

- 修改参数
- 删除步骤
- 添加步骤
- 重新排序
- 查看差异

### 6. Executor

真正执行步骤，但只调用 Host API，不直接操作 UI。

---

## 七、语义拆解的重点

语义拆解是第一优先级。

应按以下层次进行：

### 1. 意图识别

判断用户属于哪类任务，例如：

- 播放控制
- 歌单管理
- 收藏管理
- 批量整理
- 组合任务

### 2. 实体抽取

提取：

- 歌曲名
- 歌手名
- 专辑名
- 歌单名
- 数量
- 时间范围
- 风格/情绪
- 筛选条件

### 3. 指代消解

例如：

- “这首歌” = 当前播放
- “那个歌单” = 当前选中歌单
- “刚才那些” = 上一轮搜索结果

### 4. 歧义识别

当存在多种可能解释时，不应强行执行，应：

- 优先用上下文补全
- 无法补全则让用户确认

### 5. 归一化输出

最终应输出结构化结果，而不是自由文本说明。

示例：

```json
{
  "intent": "add_tracks_to_playlist",
  "entities": {
    "playlistName": "夜跑歌单",
    "artist": "周杰伦",
    "limit": 20,
    "source": "recent_top_played"
  },
  "ambiguities": [],
  "needsConfirmation": true
}
```

## 八、计划纠错机制

必须支持“AI 先生成流程，用户再修正”。

推荐分两层：

### 1. 语义确认

向用户展示 AI 理解到的目标和步骤。

例如：

```text
我理解你的需求是：
1. 创建歌单“夜跑歌单”
2. 获取最近常听的 20 首歌曲
3. 将这些歌曲加入歌单
4. 开始播放该歌单
```

### 2. 执行确认

在真正修改数据之前做最终确认。

例如：

```text
即将执行：
- 创建歌单：夜跑歌单
- 添加歌曲：20 首
- 自动播放：是
是否继续？
```

### 用户可修改的内容

- 歌单名
- 数量
- 筛选条件
- 是否自动播放
- 删除某一步
- 添加一步
- 更换目标对象

推荐让用户修改结构化步骤，而不是仅修改表面文字。

## 九、Skill 机制设计

Agent 可以支持 Skill，而且非常适合作为“运行规则包”。

Skill 的定位不是工具本身，而是：

- 语义规则
- 规划规则
- 执行规则

### Skill 可以作用于三层

#### 1. 语义 Skill

用于：

- 口语映射
- 指代补全
- 常见别名纠正

#### 2. 规划 Skill

用于：

- 指导任务如何拆步
- 指定默认计划模板
- 规定某类操作的顺序

#### 3. 执行 Skill

用于：

- 自动去重
- 幂等策略
- 风险限制
- 确认规则

### Skill 与 Tool 的区别

- Tool：真正执行动作
- Skill：指导怎么理解和怎么执行

可理解为：

```text
Tool = 手脚
Skill = 经验和规矩
```

## 十、上下文管理建议

不要简单堆聊天记录。

建议维护三层记忆：

### 1. 短期记忆

- 最近几轮用户消息
- 最近几轮 AI 消息

### 2. 工作记忆

当前任务状态，例如：

- 当前目标
- 当前歌单名
- 候选歌曲列表
- 当前执行到哪一步
- 用户是否已确认

### 3. 长期记忆

用户偏好，例如：

- 常用歌单命名风格
- 常听歌手
- 是否默认自动播放
- 常用筛选习惯

对这个项目来说，最关键的是“工作记忆”。

## 十一、RAG 的定位

RAG 是 Retrieval-Augmented Generation，即“检索增强生成”。

含义是：

```text
先检索资料，再基于资料生成回答
```

### 在当前项目中的适用范围

RAG 更适合：

- 帮助文档问答
- Skill 规则检索
- 配置说明问答
- 音乐库元数据说明

### 当前阶段不应把 RAG 当成主引擎

当前项目第一阶段更重要的是：

- 语义拆解
- 结构化计划
- 用户纠错
- 工具执行

优先级建议：

```text
语义拆解 > 计划与纠错 > 工具执行 > 上下文管理 > 工程稳定性 > RAG
```

---

## 十二、LangGraph 的定位

LangGraph 适合承担：

- 多步流程编排
- 有状态执行
- 人工确认中断
- 执行恢复
- 工具调用编排

不建议直接把 LangGraph 硬塞进 Qt 主进程。

更推荐：

```text
Qt 软件内部入口 + LangGraph Sidecar + Host API
```

第一阶段也可以先不做复杂图编排，而是先实现：

- 语义解析
- 结构化计划
- 可编辑计划
- 顺序执行器

后面再逐步演进到完整的 LangGraph 状态图。

---

## 十三、代码与工程组织建议

建议放在同一个仓库中开发，但分目录维护。

推荐结构：

```text
your-music-app/
  app/                 # Qt 主程序
  plugins/             # Qt 插件
  agent/               # Agent 后端
  shared/              # 协议、skill、schema、prompt
```

或者：

```text
MyPlayer/
  apps/
    desktop-qt/
    ai-agent/
  plugins/
    ai-assistant/
  shared/
    protocol/
    prompts/
    skills/
```

原则：

- 同仓库，方便联调和版本同步
- 分目录，避免代码耦合混乱
- Agent 后端可以单独运行和调试

---

## 十四、推荐的第一阶段范围

第一版不应追求“万能 Agent”，而应聚焦高频、可控场景。

建议先支持这些动作：

- 搜歌
- 播放指定歌曲
- 获取当前播放歌曲
- 创建歌单
- 批量向歌单加歌
- 收藏/取消收藏
- 查询最近播放/最常播放

建议优先支持这几类用户任务：

- “播放晴天”
- “把这首歌加入喜欢”
- “创建一个夜跑歌单”
- “把最近常听的 20 首加到夜跑歌单里”
- “创建一个学习歌单并开始播放”

---

## 十五、第一阶段开发优先级

推荐优先顺序：

1. 定义 Host API
2. 定义语义槽位和意图枚举
3. 做结构化语义拆解
4. 做结构化计划生成
5. 做计划预览与用户确认
6. 做执行器
7. 做上下文管理
8. 再考虑 LangGraph 深度编排
9. 最后补充 RAG

---

## 十六、给后续模型或开发者的简要提示

如果继续基于本项目扩展 Agent，请遵循以下原则：

- 把 AI 当成“规划器”，不要当成“点按钮的脚本”
- 所有操作必须通过主程序定义好的 Host API 完成
- 优先提升语义拆解稳定性，而不是先堆 RAG
- 复杂任务必须先生成结构化计划
- 计划必须允许用户修改
- 批量和高风险操作必须确认
- 上下文要结构化管理，不要无限堆历史消息
- Skill 作为规则包参与语义、规划和执行
- 通信优先使用 WebSocket + JSON
- 前端入口在 Qt 软件内，Agent 可以是独立 sidecar 进程

---

## 十七、一句话总结

目标不是做一个“会聊天的音乐 AI”，而是做一个：

**能把自然语言稳定拆成结构化意图，生成可编辑流程，并通过主程序业务 API 安全执行的 Qt 音乐软件 Agent。**
