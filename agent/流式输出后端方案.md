# Agent 后端流式输出方案

## 1. 目标

在现有最小聊天 Agent 后端基础上增加流式输出能力，让客户端能够在模型生成过程中持续收到文本增量，而不是只能等到完整回复结束后一次性展示。

本次改造目标：

- 保持现有 WebSocket 聊天入口不变
- 增加流式协议消息
- 保持 `assistant_final` 作为最终结果消息
- 会话仍然只在整轮完成后落库
- 对 Qt 客户端保持低接入成本

---

## 2. 协议设计

当前协议新增两类服务端消息：

- `assistant_start`
- `assistant_chunk`

保留原有：

- `session_ready`
- `user_message`
- `assistant_final`
- `error`

### 2.1 assistant_start

```json
{
  "type": "assistant_start",
  "sessionId": "uuid",
  "requestId": "req-1"
}
```

含义：

- 标记这轮 assistant 回复正式开始
- 允许客户端先创建一条空消息

### 2.2 assistant_chunk

```json
{
  "type": "assistant_chunk",
  "sessionId": "uuid",
  "requestId": "req-1",
  "delta": "你好"
}
```

含义：

- `delta` 是这次增量文本
- 客户端应把它追加到同一个 assistant 消息尾部

### 2.3 assistant_final

```json
{
  "type": "assistant_final",
  "sessionId": "uuid",
  "requestId": "req-1",
  "content": "你好，我是一个助手。"
}
```

含义：

- 这轮回复已经完整结束
- `content` 是服务端聚合后的最终文本
- 客户端可用 `content` 校准本地拼接结果

### 2.4 error

流式场景下仍然沿用：

```json
{
  "type": "error",
  "sessionId": "uuid",
  "requestId": "req-1",
  "code": "stream_interrupted",
  "message": "stream failed"
}
```

如果发生在已输出部分 chunk 之后：

- 客户端不应删除已显示文本
- 应把该消息标记为“生成中断”

## 3. 后端实现思路

## 3.1 模型层

在 `llm_client.py` 中新增：

- `complete(messages)`：非流式，返回完整字符串
- `stream_complete(messages)`：流式，返回异步增量文本序列

实现方式：

- OpenAI-compatible 模型调用使用 `stream=True`
- 持续读取每个 chunk 的 `delta.content`
- 对流式内容不做 `strip()`，避免丢失空格和换行

---

## 3.2 Agent 层

`ChatAgent` 保留原本的多轮上下文管理职责，但拆成更适合流式的结构：

- `respond()`：普通一次性回复
- `stream_response()`：流式回复
- `persist_turn()`：统一保存整轮对话

上下文生成改为：

- 先通过 LangGraph 读取会话历史并拼装模型输入
- 再由模型层负责流式输出

### 为什么不把每个 token 放进 LangGraph

原因：

- token 级状态更新没有必要放进图编排
- 会让图状态过于频繁变化
- server 层更适合承担 WebSocket 增量转发

因此职责边界是：

- `LangGraph`：负责准备这轮请求所需上下文
- `server.py`：负责把流式增量发给客户端

---

## 3.3 会话持久化策略

会话存储仍然采用：

- 只在整轮成功完成后写入
- 不对每个 chunk 持久化

这样做的好处：

- 减少频繁写入
- 会话结构保持简单
- 中途流式失败时不会留下半截 assistant 消息

当前行为：

- 完整流式结束：写入 user + assistant
- 仅部分 chunk 后中断：不写入这轮 assistant
- 若流式在开始前失败：回退到非流式 `complete()`

---

## 4. 服务端时序

一轮请求的服务端流程如下：

```text
receive user_message
-> validate
-> send assistant_start
-> stream model deltas
-> send assistant_chunk repeatedly
-> collect full response text
-> persist_turn
-> send assistant_final
```

### 异常处理

#### 情况 1：流式开始前失败

策略：

- 回退到 `complete()`
- 仍然按 chunk + final 的方式回给客户端

#### 情况 2：已经输出部分 chunk 后失败

策略：

- 发送 `error`
- 不持久化这轮 assistant 消息

## 6. 对客户端的影响

客户端不需要改连接方式，也不需要改发送格式。

唯一需要新增的是：

- 识别 `assistant_start`
- 识别 `assistant_chunk`
- 对同一个 `requestId` 的消息做增量拼接

也就是说，客户端发送的 `user_message` 格式不变，服务端响应从原来的“一条 final”变成：

```text
assistant_start
assistant_chunk...
assistant_chunk...
assistant_final
```

---

## 7. 当前限制

当前流式实现仍然是“单轮串行回复”：

- 同一个 WebSocket 连接上不建议并发发送多条问题
- 目前没有取消生成接口
- 还没有做断线续流

这不影响第一阶段 Qt 对接，因为当前最核心的是把流式展示链路跑通。

---

## 9. 一句话结论

当前流式输出方案的核心原则是：

**服务端边生成边通过 `assistant_chunk` 推送增量文本，客户端边收边显示，但最终仍以 `assistant_final` 作为完整结果和落库依据。**
